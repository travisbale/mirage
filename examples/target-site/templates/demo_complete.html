AiTM demo complete. Your session was successfully captured by the Mirage proxy.

What just happened?

You visited a phishing URL that loaded the real Vault login page through a reverse proxy. Everything you saw — the login form, the MFA prompt, the dashboard — was served by the legitimate Vault server. The proxy sat in the middle, transparently capturing your credentials and session tokens as they passed through.

This is an Adversary-in-the-Middle (AiTM) attack. Unlike traditional phishing which uses a fake login page, AiTM proxies the real site so the victim sees genuine content, valid TLS, and working MFA. The attacker captures the post-MFA session token, which can be replayed to access the victim's account without needing the password or MFA code again.

What was captured?

Run mirage sessions list --completed to see the captured session, including:

  • Email address and password
  • MFA verification code
  • Session cookies (replayable for account access)
  • Browser fingerprint and bot detection telemetry

Export the captured cookies with mirage sessions export <id> --out cookies.json and import them into a browser extension like StorageAce to access the victim's account.

Why did I have to log in again?

After capturing your session, the proxy redirected you here — the real Vault site. Because the session cookie was set on the phishing domain (not the real domain), your browser doesn't have a valid session for this site. In a real attack against a service like Google or Microsoft 365, the victim would typically already be logged in on the real domain, so the redirect would land on a page they can access normally.
